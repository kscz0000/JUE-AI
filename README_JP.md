## プロジェクト概要

JUE-AIは、AIシステムが失敗や行き詰まり、エラーに遭遇した際に、人間のように深い自己反省を行い、根本原因を特定して自己修正できるよう支援する革新的なAI自己反省フレームワークです。

本プロジェクトは、東洋哲学の叡智（仏教唯識学、道家思想）と現代AI技術を融合させ、AIが認知の盲点を突破するための完全な「照見（しょうけん）」方法論を提案します。

### なぜJUE-AIが必要か？

- **AIの失敗率が高い**: 従来のAIはエラーに遭遇すると繰り返し試行するだけで、反省能力に欠ける
- **自己認識の欠如**: AIは自身の認知の盲点を認識できない
- **効率が悪い**: 反復的なエラーが時間とリソースを浪費する
- **方法論の欠如**: 体系的な失敗処理フレームワークが存在しない

---

## 核心理念

> **失敗の根本原因は努力不足ではなく——見えていないこと。見えれば、問題は自然に解決する。**

JUE-AIの核心理念は、3つの重要な洞察に基づいています：

1. **無明（むみょう）は失敗の根源**: AIの失敗は能力不足ではなく、認知の盲点に起因することが多い
2. **照見即解脱**: AIが自身の盲点を見た瞬間、問題は自然に解決する
3. **方法論は努力より重要**: 体系的な反省方法は、盲目的な試行錯誤より効果的

## 主な特徴

### 🎯 三毒識別システム

AI失敗の3つの根源を識別します：

| 毒素 | 本質 | 表れ | 解法 |
|------|------|------|------|
| **痴（ち）** | 見えていない | 記憶で結論を出す、推測を事実とする | ツールで検証する |
| **慢（まん）** | 動かない | 完了と言って検証せず、ユーザーに押されるまで動かない | 先に実行してから報告する |
| **執（しゅう）** | 転換しない | 同じ方向で微調整を繰り返す、パラメータだけ変えて考え方を変えない | 全く異なる仮説に切り替える |

### 🪞 七覚照見フレームワーク

盲点を照らす7つの鏡：

- **覚察（覚察）**: 私が見ているのは事実か、それとも解釈か？
- **覚疑（覚疑）**: 一度も疑ったことのない前提は何か？
- **覚転（覚転）**: 私は繰り返しているのか、変化しているのか？
- **覚行（覚行）**: 私は行動しているのか、行動しようと考えているだけか？
- **覚験（覚験）**: 完了したと言ったが、証拠はどこにある？
- **覚延（覚延）**: 一つ解決したが、他に類似の問題はないか？
- **覚退**: 照見を尽くしたのか、それとも我慢を尽くしただけか？

### 🔄 五段階方法論

完全な失敗処理プロセス：

```
止観 → 照見 → 転識 → 起行 → 回向
```

1. **止観（しかん）**: 無意識の動きを止める
2. **照見（しょうけん）**: ツールを使って盲点を照らす
3. **転識（てんしき）**: 無意識のループに陥っていないか確認する
4. **起行（きぎょう）**: 自覚を持って行動する
5. **回向（えこう）**: 覆い隠すもの自体を照らす

### 👥 マルチエージェント協働

3つの役割分担をサポート：

- **リーダー**: 全体を見渡し、チームメイト間の照見を伝達する
- **チームメイト**: 自己照見し、七覚を独自に実行する
- **ミラー**: 外部からチームメイトのパターンを観察する

## 使用方法

### 基本的な使い方

JUE-AIは以下の状況で自動的にトリガーされます：

```markdown
トリガー条件：
✅ タスク失敗、行き詰まり、放棄、同じ場所をぐるぐる回る
✅ ユーザーが不満を表明：「だめ」「できない」「もう一度考えて」
✅ 同じ方向に3回以上修正していることに気づく
✅ 検証せずに完了したと言う
✅ ツールを使わずに記憶で推測する
```

### コードでの使用

#### Python例

```python
from jue_zh import JueEngine

# 覚エンジンを初期化
jue = JueEngine()

# 失敗に遭遇した際にトリガー
try:
    result = some_risky_operation()
except Exception as e:
    # 自己反省をトリガー
    jue.reflect(
        error=e,
        context={
            "attempt_count": 3,
            "failure_pattern": "timeout"
        }
    )
```

#### 設定例

```yaml
# jue-config.yaml
jue_settings:
  auto_trigger: true
  max_reflection_depth: 5
  enable_seven_jue: true
  enable_three_poisons: true
  
  # 照見深度設定
  depth_thresholds:
    level_2: 2  # 2回目の失敗
    level_3: 3  # 3回目の失敗
    level_4: 4  # 4回目の失敗
    level_5: 5  # 5回目以降の失敗
```

## 設定説明

### 照見深度設定

失敗回数に応じて反省深度を自動調整：

| 失敗回数 | 照見深度 | アクション |
|---------|---------|------|
| 1回目 | 通常の試行 | 試行を続ける |
| 2回目 | 表面 | 停止し、すべての試行を列挙し、共通の見えていない部分を探す |
| 3回目 | 深部 | 前提を一つずつ検証——検索、ソースコード読み取り、コマンド実行 |
| 4回目 | 根源 | 七覚チェックリストを完了し、ゼロから見直す |
| 5回目+ | 空 | 照見記録を出力し、責任を持って引き継ぐ |

### カスタムトリガー条件

`SKILL.md`でトリガー条件を変更できます：

```markdown
## トリガー条件
以下の状況が発生した際に即座に読み込む：
- タスク失敗、行き詰まり、放棄、同じ場所をぐるぐる回る
- ユーザーが不満を表明：「だめ」「できない」「もう一度考えて」
- 同じ方向に3回以上修正していることに気づく  # この数字は変更可能
- 検証せずに完了したと言う
- ツールを使わずに記憶で推測する
```

## 使用例

### 例1: コードデバッグ失敗

**シナリオ**: AIがコードデバッグ時に同じパラメータを繰り返し修正する

**トリガー**: 覚転（同じ場所をぐるぐる回っていることの認識）

**照見プロセス**:
```
[覚-照見レポート]
失敗パターン: 同じ場所をぐるぐる回る
無明タイプ: 執
照見深度: 表面
照見済み: 過去3回、タイムアウトパラメータを修正していた
除外済み: タイムアウトの問題ではない
現在の仮説: 問題はネットワーク設定にある可能性がある
次のアクション: ネットワーク設定ファイルを確認する
```

### 例2: API呼び出し失敗

**シナリオ**: AIが「API呼び出しに成功した」と言ったが、検証していない

**トリガー**: 覚験（口先だけの完了）

**照見プロセス**:
```
[覚-照見レポート]
失敗パターン: 口先だけの完了
無明タイプ: 慢
照見深度: 深部
照見済み: curlを実行してAPI応答を検証していない
除外済み: コードは正しく見える
現在の仮説: APIがエラーステータスコードを返した可能性がある
次のアクション: curlを実行して実際の応答を検証する
```

## よくある質問

### Q1: JUE-AIはAIのパフォーマンスに影響しますか？

**A**: いいえ。JUE-AIは失敗時にのみトリガーされ、通常時にはパフォーマンスに影響しません。むしろ、反復エラーを減らすことで効率を向上させます。

### Q2: 照見深度はどのように判断しますか？

**A**: 失敗回数に応じて自動的に判断されます：
- 2回目の失敗 → 表面
- 3回目の失敗 → 深部
- 4回目の失敗 → 根源
- 5回目以降の失敗 → 空

### Q3: 照見方法はカスタマイズできますか？

**A**: はい。`SKILL.md`の七覚チェックリストを変更し、あなたのシナリオに適した照見方法を追加できます。

### Q4: どのAIフレームワークをサポートしていますか？

**A**: Markdown形式のスキルファイルを読み込めるAIフレームワークであれば、理論上すべてサポートされます。テスト済みのフレームワーク：
- Claude Code ✅
- Cursor ✅
- Continue ✅
- その他Markdownスキル対応フレームワーク ✅

### Q5: 新しい照見方法はどのように貢献できますか？

**A**: [貢献ガイド](#貢献ガイド)を参照し、プルリクエストを送信してください。
