# 觉的方法论 — 止观照见五步

> 此参考文件提供完整的方法论流程。当需要按步骤执行止观→照见→转识→起行→回向时读取。

每次失败或卡壳后按以下 5 步执行。代码、研究、写作、规划都适用。

这套方法论的核心不是「做什么」，而是「看见什么」。每一步都在扩展你的视野，照亮之前看不见的区域。

## 第一步：止观 — 停止无意识的运动

> 知止而后有定，定而后能静，静而后能安，安而后能虑，虑而后能得。
> — 大学

不是停止努力，是停止**不知道自己在做什么**的努力。

当你发现自己在快速尝试各种方案时，停下来。这种「快速尝试」本身往往就是问题——你在用速度代替方向。

**操作**：
1. 列出过去所有尝试
2. 在每个旁边写下：它的前提假设是什么？
3. 画出共同假设——那就是你的盲区
4. 如果所有尝试共享同一前提假设，你找到了问题所在

**检验**：你能否用一句话说清「我已照见什么，还没照见什么」？不能 = 你还没真正停下来。

## 第二步：照见 — 用工具照亮盲区

> 吾日三省吾身。
> — 论语

照见有五个维度，按顺序执行：

### 维度一：逐字读失败信号
报错信息、拒绝原因、空结果、用户不满——不是扫一眼，是逐字读。你跳过的那行报错里可能就有答案。你觉得「不重要」的那个警告可能就是根因。

### 维度二：主动搜索
不要靠记忆。让工具告诉你：
- 代码 → 搜索**完整**报错信息（完整的，不是你截取的）
- 研究 → 搜索多个不同关键词角度
- API/工具 → 搜索官方文档 + GitHub Issues

### 维度三：读原始材料
不是读记忆，不是读摘要，是读**原始来源**：
- 代码 → 出错位置上下文 50 行
- API → 官方文档原文（不是你记得的文档）
- 研究 → 原始来源（不是二手引用）

### 维度四：验证前置假设
你认为成立的所有条件，哪个没用工具验证过？全部确认：
- 代码 → 版本（`node -v`）、路径（`which`）、权限（`ls -la`）、依赖
- 数据 → 字段名、格式、值域
- 逻辑 → 边界情况、异常路径、并发

### 维度五：反转假设
如果你一直假设「问题在 A」，现在假设「问题不在 A」，从对立方向看。

这是最强大的照见工具——你的注意力一旦锁定某个方向，所有证据都会被你解读为支持那个方向。反转假设打破这个锁定。

> 维度 1-4 完成前不要向用户提问。带着证据提问。

## 第三步：转识 — 检查是否在无意识中循环

> 回头是岸。
> — 禅语

问自己四个问题：

1. **我在重复同一思路的变体吗？** 方向不变只是参数不同 = 循环
2. **我在看表面症状还是深层原因？** 报错是症状，写法/设计/配置才是原因
3. **我该用工具却在用记忆吗？** 有搜索不搜、有文件不读、有命令不跑
4. **我检查了最简单的可能性吗？** 错别字、格式错误、前提条件不满足

任何一个「是」→ 你找到了遮蔽物。移除它。

---

## 第四步：起行 — 带着觉知行动

> 知者行之始，行者知之成。
> — 王阳明

新方案必须满足三个条件：

1. **本质不同**：不共享旧方案的前提假设
2. **可验证**：有明确的「如果对了看到什么，错了看到什么」
3. **失败也有价值**：即使不成功，也能排除一种可能，照见新信息

不满足这三个条件的方案不要执行。它只会消耗时间，不会照见新东西。

---

## 第五步：回向 — 照见遮蔽本身

> 悟已往之不谏，知来者之可追。
> — 归去来辞

这一步是整个方法论的灵魂。

普通的复盘问「哪里做得不好」→ 优化执行。
觉的复盘问「**什么遮住了我的眼睛**」→ 照见无明本身。

**操作**：
1. 哪个方案解决了？
2. 之前的方案为什么没解决？它们的共同盲区是什么？
3. **什么遮住了我的眼睛？** 是记忆（痴）、惰性（慢）、还是固执（执）？
4. 同类问题是否还存在？
5. 如何防止下次被同一种遮蔽困住？

回向不是自我批评，是自我照见。不是「我做错了什么」，而是「我没看见什么」。这个区别决定了你下次是否还会被同一种遮蔽困住。

---

## 五步的节奏

五步不一定线性执行。它们像呼吸：

- 止观 = 吸气（停下，收集信息）
- 照见 = 屏息（深入观察）
- 转识 = 呼气（释放错误假设）
- 起行 = 再次吸气（带着新理解行动）
- 回向 = 长呼一口气（照见整个过程）

有时第二步就照见了答案，有时需要循环多次。关键不是步骤，是**觉知是否在场**。
